# Workflow (как я работаю над проектом)

## 1) Создаю задачу (GitHub Issue)
1. Создать Issue с понятным названием: `GET /api/afisha/films`
2. Добавить описание: что должно работать
3. Вставить DoD чеклист (ниже)
4. Добавить Issue на Kanban (Backlog)

## 2) Беру задачу в работу
1. Перетащить карточку в `Next`
2. Начал кодить → в `In progress`

## 3) Реализую
- Контроллер: только принимает/возвращает
- Сервис: вся логика
- DTO: вход/выход данных

## 4) Проверяю (Review / Test)
Перетащить в `In review / test` и пройти DoD.

### DoD для API задач
- [ ] Эндпоинт работает
- [ ] Проверено через Postman
- [ ] Возвращаются правильные статусы (200/400/404/409 где нужно)
- [ ] Нет ошибок в логах
- [ ] Логика в сервисе (контроллер тонкий)

### DoD для инфраструктуры (настройки/модули)
- [ ] Проект запускается
- [ ] Нет ошибок в логах
- [ ] Настройки задокументированы в README / .env.example

## 5) Коммиты (Conventional Commits)
Формат: `type: message`

Разрешённые типы:
- feat: новая функциональность
- fix: исправление бага
- refactor: переработка без изменения поведения
- docs: документация
- chore: настройка/зависимости/скрипты
- test: тесты

Примеры:
- feat: implement GET /api/afisha/films
- fix: prevent double booking seats
- chore: add mongodb connection
- refactor: move booking logic to service
- docs: add run instructions

## 6) Завершение задачи
1. Если DoD пройден → перенос в `Done`
2. Закрыть Issue (если не закрывается автоматически)
